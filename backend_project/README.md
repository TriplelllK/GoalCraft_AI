## 🎯 О проекте

**GoalCraft AI** — интеллектуальная платформа для управления целями персонала в нефтегазовой компании. Решает ключевую проблему HR: ежеквартально тысячи сотрудников формулируют цели, но >60% из них не соответствуют методологии SMART/OKR.

### Что делает система:

| Функция | Описание |
|---------|----------|
| 🔍 **Оценка цели** | Мгновенный SMART-анализ по 5 критериям (S, M, A, R, T) с детальными рекомендациями |
| ✏️ **Переформулировка** | AI перепишет слабую цель в сильную с сохранением смысла |
| 🧠 **Генерация** | 3–5 релевантных целей на основе роли, KPI, стратегии компании и RAG-контекста |
| 📊 **Пакетная оценка** | Batch-анализ всех целей сотрудника: веса, дубликаты, баланс типов |
| 🔗 **Каскадирование** | Автоматическая декомпозиция целей руководителя на подчинённых |
| 📈 **Дашборд** | Аналитика по подразделениям: SMART-индексы, зрелость, ранжирование |
| 🔔 **Alert Manager** | Проактивные уведомления о проблемах (пустые цели, низкие баллы, дисбаланс весов) |

## 🧮 SMART-оценка целей — как это работает

### Алгоритм: детерминированный скоринг по 5 критериям

Каждая цель проходит через 5 независимых анализаторов, каждый возвращает балл **0.10–0.95**:

#### S — Specific (Конкретность)

```
+0.35  содержит глагол действия (внедрить, увеличить, разработать, ...)
+0.25  конкретный объект (KPI, SLA, процесс, система, ...)
+0.10  длина ≥ 5 слов
+0.10  длина ≥ 8 слов (бонус за детализацию)
+0.15  механизм достижения — «за счёт ...»
−0.20  размытая лексика (улучшить, оптимизировать без контекста)
−0.15  перегруженность (>30 слов или ≥3 перечисления)
```

#### M — Measurable (Измеримость)

```
0.92  найдена метрика: %, SLA, количество, срок, доля, уровень, ...
0.55  метрика нереалистична (>500%, >10000 единиц)
0.30  метрика не найдена
```

#### A — Achievable (Достижимость) — с историческим анализом (F-20)

```
1. Поиск похожих целей за прошлые кварталы (Jaccard ≥ 0.15)
2. Оценка SMART-индексов исторических целей
3. Если |текущий − средний_исторический| ≥ 0.3 → нереалистичная цель
4. Confidence = min(1.0, кол-во_похожих × 0.25)
```

#### R — Relevant (Релевантность) — с RAG-контекстом

```
Поиск по 60+ HR-доменным ключевым словам
+0.05  совпадение с ролью сотрудника
+0.05  совпадение с подразделением
+0.10  совпадение с RAG-чанком из стратегии/ВНД
```

#### T — Time-bound (Ограниченность во времени)

```
0.91  найден дедлайн: дата, квартал (Q1–Q4), «ежемесячно», «до ...»
0.28  дедлайн не найден
```

### Итоговый SMART-индекс

```
overall_score = (S + M + A + R + T) / 5
```

### Классификация типа цели

| Тип | Условие | Пример |
|-----|---------|--------|
| **impact-based** | Глагол изменения + метрика | «Снизить текучесть на 15% к Q3» |
| **output-based** | Глагол результата | «Внедрить систему KPI до 30.06» |
| **activity-based** | Остальное | «Проводить обучение» |

### Уровень стратегического выравнивания

| Уровень | Условие |
|---------|---------|
| **strategic** | Стратегические ключевые слова + конкретность + RAG-чанк из стратегии |
| **functional** | KPI/обучение/координация |
| **operational** | По умолчанию |

## 🔎 RAG-система — обучение и поиск по корпоративным документам

### Зачем RAG?

Система должна понимать контекст компании: стратегию, ВНД, KPI-каталог, цели руководителей. RAG позволяет обогащать оценку и генерацию целей **реальными корпоративными данными**.

### Архитектура RAG-пайплайна

```
Документы (стратегии, ВНД, KPI)
          │
          ▼
┌─────────────────────────┐
│  Sentence-aware Chunking │  ← разбиение с учётом границ предложений
│  max=300 chars, overlap=50│
└────────────┬────────────┘
             │
     ┌───────▼───────┐
     │   Indexing     │
     │ • N-gram hash  │  ← 512-мерный вектор (SHA-256 feature hashing)
     │ • BM25 index   │  ← инвертированный индекс для точного поиска
     │ • Metadata     │  ← dept_id, doc_type, keywords
     └───────┬───────┘
             │
     ┌───────▼───────────────────┐
     │   Hybrid Search           │
     │                           │
     │ score = cosine × 0.40     │  ← семантическое сходство
     │       + BM25   × 0.35     │  ← точное совпадение терминов
     │       + keyword × 0.15    │  ← совпадение ключевых слов
     │       + doc_type× 0.10    │  ← бонус по типу документа
     └───────────────────────────┘
```

### Sentence-aware Chunking (300 символов, overlap 50)

В отличие от наивного разбиения по символам, наш чанкер:
1. **Разбивает по границам предложений** ([.!?;] + пробел)
2. **Накапливает предложения** до лимита 300 символов
3. **Overlap-буфер** — последнее предложение переходит в следующий чанк
4. **Слияние маленьких хвостов** — если остаток < порога, сливает с предыдущим чанком

### N-gram Feature Hashing (512 измерений)

Вместо тяжёлых трансформер-эмбеддингов мы используем **лёгкий, но эффективный** метод:

```
Для каждого токена текста:
  unigrams (вес 1.0) + bigrams (вес 0.5) + char-trigrams (вес 0.5)
       │
       ▼
  SHA-256 → 2 слота в 512-мерном векторе
  Знак (+1/−1) определяется битами хеша
       │
       ▼
  L2-нормализация → единичный вектор
```

**Почему не трансформеры?** — Zero dependencies, работает offline, ~1000× быстрее BERT, а в комбинации с BM25 даёт сопоставимое качество для доменного поиска на русском языке.

### BM25 Scoring (k₁=1.5, b=0.75)

```
BM25(q, d) = Σ IDF(t) × (TF(t,d) × (k₁ + 1)) / (TF(t,d) + k₁ × (1 − b + b × |d|/avgdl))
```

Бинарный TF (0/1) — достаточно для коротких чанков (≤300 символов).

### Hybrid Search Formula

```
final_score = cosine_sim × 0.40        # семантика (n-gram vectors)
            + min(1, BM25/10) × 0.35   # точное совпадение (BM25)
            + min(0.3, kw × 0.1) × 0.15 # keyword boost
            + doc_type_bonus × 0.10     # приоритет по типу

doc_type_bonus: strategy=0.05, kpi=0.04, manager_goal=0.03, vnd=0.02
```

Результаты фильтруются по `department_id` для релевантности подразделению.

## 🤖 LLM-интеграция — GPT-4o-mini с graceful degradation

### Принцип: «LLM-first, rules-fallback»

```
┌──────────────────┐     ┌──────────────────┐
│  OpenAI API      │────▶│  GPT-4o-mini     │
│  OPENAI_API_KEY  │     │  • temp: 0.3–0.8 │
└────────┬─────────┘     └──────────────────┘
         │ Нет ключа / ошибка API
         ▼
┌──────────────────┐
│  Rule-based      │  ← Детерминированные правила (300+ строк)
│  fallback        │  ← Всегда работает, даже offline
└──────────────────┘
```

### 3 LLM-операции

| Операция | Temperature | Что делает | Fallback |
|----------|-------------|------------|----------|
| **Генерация** | 0.8 | 3–5 целей с RAG-контекстом (top-5 чанков) | 15+ шаблонов по ролям + RAG-ключевые слова |
| **Переформулировка** | 0.5 | Rewrite слабой цели + RAG-контекст (top-2) | Шаблонный rewriter: глагол + метрика + дедлайн + механизм |
| **OKR-маппинг** | 0.3 | Objective + Key Results + ambition/transparency | Keyword→objective маппинг, heuristic-scoring |

### Промпты

Все промпты **на русском языке**, содержат:
- Определения SMART-критериев
- Правила форматирования (7 для генерации, 8 для rewrite)
- Инструкции использовать числовые метрики из RAG-контекста
- Требование указывать конкретный год и квартал

### Auto-quality check при генерации

```python
# Каждая сгенерированная LLM-цель проходит SMART-оценку
# Если overall_score < 0.7 → автоматический rewrite
for goal in llm_generated:
    score = evaluate(goal)
    if score < 0.7:
        goal = rewrite(goal)  # Авто-улучшение
```

## 📊 Аналитический дашборд

### 5 страниц интерактивной аналитики

#### 1. Оценка целей (`/evaluate`)
- SMART-breakdown с цветовыми индикаторами по каждому критерию
- Рекомендации по улучшению
- AI-переформулировка в один клик
- Проверка достижимости (F-20) с историческим анализом
- OKR-маппинг с ambition/transparency скорами
- Пакетная оценка всех целей сотрудника

#### 2. Генерация целей (`/generate`)
- Карточки сгенерированных целей с SMART-баллом
- Alignment level, goal type, rationale
- Источник (RAG-документ) для прозрачности

#### 3. Каскадирование (`/cascade`)
- Цели руководителя → декомпозиция на подчинённых
- Визуализация связи manager_goal → subordinate_goals

#### 4. Дашборд (`/dashboard`)
- **Grouped Bar Chart**: SMART-индекс vs Strategic share по подразделениям (Recharts)
- **Pie Chart**: Распределение уровней зрелости
- **Ranking Table**: Все подразделения с цветовой кодировкой
- Детальная панель по каждому подразделению

#### 5. Зрелость (`/maturity`)
- **Maturity Index** = `avg_SMART × 0.45 + strategic_share × 0.35 + coverage × 0.15 + 0.05`
- Круговой индикатор зрелости
- SMART-распределение, типы целей, alignment breakdown

### Формула индекса зрелости

```
maturity_index = min(1.0,
    avg_smart_index × 0.45          # Качество целей
  + strategic_goal_share × 0.35     # Доля стратегических целей
  + min(0.15, employees × 0.05)     # Покрытие целеполаганием
  + 0.05                            # Базовый бонус
)

Уровни:
  ≥ 0.80 → «продвинутый»
  ≥ 0.60 → «зрелый»
  ≥ 0.40 → «развивающийся»
  < 0.40 → «начальный»
```

### Alert Manager (уведомления)

## 🚀 Инструкция по запуску

### Вариант 1: Локальный запуск (Demo — без Docker, без БД)

```bash
# 1. Клонируйте репозиторий
git clone https://github.com/TriplelllK/GoalCraft_AI.git
cd GoalCraft_AI

# 2. Создайте виртуальное окружение
python -m venv .venv

# Windows:
.venv\Scripts\activate
# Linux/macOS:
source .venv/bin/activate

# 3. Установите зависимости
pip install -r requirements.txt

# 4. (Опционально) Подключите LLM
# Windows PowerShell:
$env:OPENAI_API_KEY = "sk-proj-ваш-ключ"
# Linux/macOS:
export OPENAI_API_KEY="sk-proj-ваш-ключ"

# 5. Запустите бэкенд
cd backend_project
uvicorn app.main:app --host 0.0.0.0 --port 8899

# 6. Проверьте работу
# Swagger UI: http://localhost:8899/docs
# Health:     http://localhost:8899/health
```

